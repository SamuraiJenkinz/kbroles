## Accomplishments

- `getCca()` returns a module-level `ConfidentialClientApplication` singleton lazy-initialised from `env().ENTRA_CLIENT_ID` / `ENTRA_TENANT_ID` / `ENTRA_CLIENT_SECRET`. Pitfall 11 compliant: reused across all future /api/login + /api/auth/callback requests.
- `__resetCcaForTests()` hook nulls the cache so each test can stub a fresh env without cross-test leakage.
- `SessionData` interface declares `user?: {oid, email, name, roles[]}` — decoded claims only, never raw idToken (matches xmcp Flask `session["user"] = result.get("id_token_claims")`).
- `getSessionOptions()` is a **function** (not a top-level const) reading `env().SESSION_SECRET` lazily — Pitfall 2 compliant. Cookie defaults: `kb_session`, httpOnly, sameSite=lax, path=/, maxAge 8h, secure in production.
- `getSession()` / `saveSession()` / `clearSession()` wrap iron-session with Next.js 15 async `cookies()` correctness (Pitfall 10). All three accept an optional pre-awaited cookieStore for test injection.
- Zero browser-MSAL imports (`@azure/msal-browser`, `@azure/msal-react`, `@microsoft/teams-js`) in either new file — Plan 06 can safely delete them without breaking Plan 02 code.

## Decisions Made

- **`import 'server-only'` omitted — JSDoc fallback used.** Vitest 3.2.4 on this Next.js 16 project cannot resolve the bare specifier `server-only`. Root cause: Next.js 16 ships the package inside `next/dist/compiled/server-only` but does not install it as a top-level `node_modules/server-only` dep, so neither Vite nor Node's resolver sees it. The plan explicitly documented this fallback. Resolution: `// server-only` JSDoc comment + file location under `src/auth/` + the imports themselves (`@azure/msal-node` requires Node fs/crypto; `next/headers` only exists at the App-Router server boundary) collectively enforce the server-only invariant at build time. If a future dep-install adds `server-only` as a resolvable package, restore the import.
- **Authority without `/v2.0` suffix.** msal-node appends `/v2.0/.well-known/openid-configuration` internally when fetching OIDC metadata. Including `/v2.0` in the authority string would produce `/v2.0/v2.0/...` and fail. Defensive test (`authority does NOT contain /v2.0`) locks the invariant.
- **SessionData.user shape = decoded claims subset only.** Chose `{oid, email, name, roles[]}` (matches xmcp `result.get("id_token_claims")` subset) over storing raw `result.idToken`. Storing JWT strings in session cookies is a double-seal anti-pattern and inflates cookie size ~3x.
- **Optional cookieStore parameter on all three session helpers.** Enables (a) caller-side `await cookies()` reuse across multiple helpers in one route handler and (b) test injection of a marker object without mocking `next/headers` at every call site.
