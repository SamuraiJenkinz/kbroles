## Decisions Made

- **@azure/msal-node version:** Plan hinted 2.x (latest 2.16.2); pnpm resolved 5.1.4 as current `latest` dist-tag. No pre-release suffix (no alpha/beta) so this is the legitimate stable release — proceeded without stopping. The MSAL Node library jumped from 2.x to 5.x during routine major releases; API for `ConfidentialClientApplication` remains the same shape the plan expects (Plan 05.1-02 consumed it successfully with identical API).
- **Phase-5 deps preserved:** Per plan's explicit anti-pattern, did NOT `pnpm remove` any of @azure/msal-browser / @azure/msal-react / @microsoft/teams-js / jose / mock-jwks. Plan 06 removes them after frontend + middleware stop referencing them.
- **SESSION_SECRET dev default >=32 chars:** Used `'dev-only-session-secret-32-chars-min-padding-xxxxxxx'` (54 chars) so iron-session AES-256-GCM key derivation doesn't throw in tests that boot with the default. Production-grade enforcement happens at iron-session's runtime, not duplicated in zod.
- **simulateSdkMissing via getter-throw not vi.resetModules:** Initial implementation used `vi.resetModules() + vi.doMock()` to simulate the "SDK package absent" path. This caused the mock to not restore cleanly for subsequent tests — the next test hit the real AWS SDK credential chain (+2.1s timeout). Replaced with `get SecretsManagerClient() { if (simulateSdkMissing) throw new Error('Cannot find module ...'); return vi.fn()... }` — the getter re-evaluates per access, so the flag flips in and out without module-cache pollution. Speed improved from 2.5s to 417ms for the 7-test file.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 — Blocking] vi.resetModules() pollution in SDK-missing test caused real-AWS calls in sibling tests**

- **Found during:** Task 2 (secrets.test.ts initial run)
- **Issue:** The initial "SDK missing" test used `vi.resetModules() + vi.doMock(throws) + dynamic import`. After the test's cleanup (`vi.doUnmock + vi.resetModules`), the vi.mock factory at the top of the file did NOT re-register for subsequent tests. The "malformed JSON" test then imported the REAL `@aws-sdk/client-secrets-manager`, hit the real AWS credential provider chain, and took 2.1 seconds to fail before the test's `try/catch` absorbed it. Cosmetically green but measurably slow + real-network on each CI run.
- **Fix:** Replaced vi.resetModules approach with a getter-based mock factory: `get SecretsManagerClient() { if (simulateSdkMissing) throw new Error('Cannot find module ...'); return vi.fn()... }`. The `simulateSdkMissing` flag is flipped per-test and the factory getter re-evaluates on each destructured-import access inside `loadSecrets()`. No module-cache manipulation needed. Sibling tests see the restored real factory immediately on next `beforeEach`.
- **Files modified:** `src/config/__tests__/secrets.test.ts` only
- **Verification:** Test file runs in 417ms instead of 2.5s. No "Could not load credentials from any providers" real-AWS stdout noise. Full suite still 592 green.
- **Committed in:** content absorbed into `e54dcb6` (see Deviation #2 below — content-correctness preserved).
