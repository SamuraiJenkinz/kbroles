- DO NOT attempt to automate Entra App Registration creation in the workflow. That's a one-time manual human step (Plan 08 documents it).
- DO NOT delete the E2E remote smoke spec. It remains the Pitfall-11 regression guard (catches middleware misconfig from a REAL deployed environment, not just the laptop). Just rename the env var + update the comment.
- DO NOT add `NSSM stop KbAssistant` commands — use `schtasks /end /tn KbAssistant` (stops the running instance) and `schtasks /run /tn KbAssistant` (starts a new instance) per Task Scheduler semantics. The Task's "Settings → Allow task to be run on demand" must be enabled — documented in Plan 08.

**REPLACE `.github/workflows/deploy.yml` entirely** with:

```yaml
name: Deploy to On-Prem Windows

on:
  push:
    branches: [main]
  workflow_dispatch: {}

env:
  NODE_VERSION: '20.x'

jobs:
  build:
    name: Build standalone bundle
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup pnpm
        uses: pnpm/action-setup@v4
        with:
          version: 9

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: ${{ env.NODE_VERSION }}
          cache: 'pnpm'

      - name: Install dependencies
        run: pnpm install --frozen-lockfile

      - name: Typecheck
        run: pnpm typecheck

      - name: Test (unit)
        run: pnpm test

      - name: Build (Next.js standalone)
        # Phase 5.1 — no NEXT_PUBLIC_ENTRA_* injection. All auth is server-
        # side (iron-session + msal-node) and reads env at request time from
        # AWS Secrets Manager on the Windows box. Build-time secrets are no
        # longer required.
        run: pnpm build

      - name: Assemble standalone bundle
        # Pitfall 3 — Next.js standalone output OMITS `public/` and
        # `.next/static/`. Must copy them in before tarring or the deployed
        # bundle returns 404 for all public assets.
        run: |
          cp -r public .next/standalone/public
          cp -r .next/static .next/standalone/.next/static

      - name: Package artifact
        run: tar -czf kbassistant-build.tar.gz -C .next/standalone .

      - name: Upload artifact
        uses: actions/upload-artifact@v4
        with:
          name: kbassistant-build
          path: kbassistant-build.tar.gz
          retention-days: 3

  deploy:
    name: Deploy to Windows box
    runs-on: [self-hosted, windows, kbassistant]
    needs: build
    environment: production
    env:
      KBASSISTANT_DEPLOY_ROOT: ${{ vars.KBASSISTANT_DEPLOY_ROOT || 'D:\kbroles\.next\standalone' }}
      KBASSISTANT_SCHEDULED_TASK: ${{ vars.KBASSISTANT_SCHEDULED_TASK || 'KbAssistant' }}
      KBASSISTANT_HEALTH_URL: ${{ vars.KBASSISTANT_HEALTH_URL || 'http://localhost:3001/api/health' }}
    steps:
      - name: Download artifact
        uses: actions/download-artifact@v4
        with:
          name: kbassistant-build

      - name: Stop Scheduled Task, swap files, start, canary
        shell: pwsh
        run: |
          $ErrorActionPreference = 'Stop'
          $deployRoot = $env:KBASSISTANT_DEPLOY_ROOT
          $taskName   = $env:KBASSISTANT_SCHEDULED_TASK
          $healthUrl  = $env:KBASSISTANT_HEALTH_URL
          $prevRoot   = "$deployRoot.prev"

          Write-Host "Stopping scheduled task $taskName..."
          schtasks /end /tn $taskName | Out-Null
          # schtasks /end returns exit 1 if the task is not currently running — tolerate.

          # Rotate: previous .prev (if any) → gone; current → .prev; new → current.
          if (Test-Path $prevRoot) {
            Write-Host "Removing stale $prevRoot..."
            Remove-Item $prevRoot -Recurse -Force
          }
          if (Test-Path $deployRoot) {
            Write-Host "Rotating $deployRoot → $prevRoot..."
            Rename-Item -Path $deployRoot -NewName ([System.IO.Path]::GetFileName($prevRoot))
          }

          Write-Host "Extracting new bundle into $deployRoot..."
          New-Item -ItemType Directory -Path $deployRoot -Force | Out-Null
          tar -xzf kbassistant-build.tar.gz -C $deployRoot

          Write-Host "Starting scheduled task $taskName..."
          schtasks /run /tn $taskName
          if ($LASTEXITCODE -ne 0) {
            throw "schtasks /run failed with exit $LASTEXITCODE"
          }

          Write-Host "Waiting 30s for service warm-up..."
          Start-Sleep -Seconds 30

          Write-Host "Canary smoke: $healthUrl"
          $response = Invoke-RestMethod -Uri $healthUrl -TimeoutSec 15
          if ($response.status -ne 'ok') {
            throw "Health check failed: $($response | ConvertTo-Json -Compress)"
          }
          Write-Host "Deploy verified: $($response | ConvertTo-Json -Compress)"

      - name: Rollback on failure
        if: failure()
        shell: pwsh
        run: |
          $deployRoot = $env:KBASSISTANT_DEPLOY_ROOT
          $taskName   = $env:KBASSISTANT_SCHEDULED_TASK
          $prevRoot   = "$deployRoot.prev"

          Write-Host "Rollback triggered. Stopping task..."
          schtasks /end /tn $taskName | Out-Null

          if (Test-Path $prevRoot) {
            if (Test-Path $deployRoot) {
              Remove-Item $deployRoot -Recurse -Force
            }
            Write-Host "Restoring $prevRoot → $deployRoot..."
            Rename-Item -Path $prevRoot -NewName ([System.IO.Path]::GetFileName($deployRoot))
            schtasks /run /tn $taskName
            Write-Host "Rollback complete. Manual investigation required."
          } else {
            Write-Host "No $prevRoot to roll back to. Task remains stopped. Manual intervention required."
          }
```

Notes on the workflow:

1. **Runner labels**: The `[self-hosted, windows, kbassistant]` tuple requires the Windows runner to be registered with the label `kbassistant`. Plan 08 documents the one-time registration. Until Plan 08's registration is complete, `workflow_dispatch` runs will queue indefinitely — user will do the registration as part of onboarding.

2. **Path parameterisation**: `KBASSISTANT_DEPLOY_ROOT`, `KBASSISTANT_SCHEDULED_TASK`, `KBASSISTANT_HEALTH_URL` come from either GitHub Repository Variables (`vars.*`) OR defaults. Default `D:\kbroles\.next\standalone` matches xmcp convention. User overrides by setting repo-level variables in GitHub Settings → Secrets and variables → Actions → Variables.

3. **Health URL port 3001**: Matches research pattern — Next.js listens on `127.0.0.1:3001` with IIS at 443 terminating TLS. If the user deploys with a different port, they override `KBASSISTANT_HEALTH_URL`.

4. **Pitfall 3 — public/.next/static copy**: Preserved from Phase-5 workflow. Without this, deployed app 404s on all static assets.

5. **No Azure OIDC federated credentials**: All Azure-specific auth scaffolding is gone.

6. **Rollback pattern**: The `if: failure()` step only runs if the Deploy step failed. It stops the task, renames `.prev` back to the primary deploy dir, and restarts. This protects against a bad build bricking the service — the previous green build is always one rename away.

7. **Canary gates on `{status:'ok'}`**: The existing `/api/health` route (from Phase 5 Plan 05-02) returns `{status:'ok'|'degraded', checks:{env,mgti}}`. `status:'ok'` requires both env parse + MGTI reachability. A `degraded` response (e.g. MGTI temporarily unreachable) FAILS the canary, which rolls back — this is the correct behaviour for a pilot.
