## Decisions Made

1. **Kept spec file name `auth-middleware-from-azure.spec.ts` as-is.** Renaming to `auth-middleware-from-prod.spec.ts` (or similar) would ripple into `playwright.config.ts` include patterns, CI artifact names, and any external references. Plan explicitly allowed it as a judgment call and flagged renaming as optional cleanup. Noted as future hygiene task below.
2. **Parameterised deploy path / task name / health URL with defaults.** Workflow works out-of-the-box on a fresh Windows runner without any GitHub Repository Variables set (defaults match xmcp convention: `D:\kbroles\.next\standalone`, `KbAssistant`, `http://localhost:3001/api/health`). User overrides only if their deploy layout differs.
3. **Tolerated non-zero exit from `schtasks /end`.** First-deploy scenario (task not yet running) would otherwise fail the step. Used `| Out-Null` to suppress output + rely on `/run` exit code as the real success signal.
4. **Health canary gates on `{status:'ok'}` strictly.** A `degraded` response (MGTI temporarily unreachable) also fails the canary and triggers rollback. Correct for pilot â€” a degraded deploy should not become the new stable. Phase 6 may relax this to `status in ('ok','degraded')` for production after observability is in place.

## User Setup Required

None introduced by this plan directly â€” however, this workflow is the **contract** that Plan 08 documents operator setup against:

- Register a GitHub self-hosted runner on the Windows box with labels `self-hosted, windows, kbassistant`.
