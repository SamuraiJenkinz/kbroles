# Phase 5.1: MMC-IT BFF Pivot (xmcp Pattern) - Research

**Researched:** 2026-04-23
**Domain:** Confidential-client Entra ID BFF auth, @azure/msal-node, iron-session, AWS Secrets Manager, Next.js on-prem Windows + NSSM + IIS
**Confidence:** HIGH (xmcp source read directly; Microsoft and iron-session docs verified)

---

## Summary

Phase 5.1 replaces the SPA+NAA architecture with the MMC-IT-blessed BFF pattern as proven by `C:/xmcp/` (Atlas Exchange Infrastructure). The pivot is a **surgical replacement** of the auth layer, not a rewrite of the application: the chat grounding, role UX, citations, source panel, and error display all remain unchanged. The recommendation is **Option (a)** â€” keep Next.js, implement auth in route handlers using `@azure/msal-node` + `iron-session` â€” because it is structurally identical to xmcp's Flask+MSAL pattern translated to Node/Next, requires no new server process, and keeps the standalone Next.js output as the sole deployable artifact.

The auth code flow pattern maps directly: `/api/login` calls `cca.getAuthCodeUrl()` (stores state + PKCE verifier in an iron-session pre-auth cookie) â†’ Entra redirects to `/api/auth/callback` â†’ callback calls `cca.acquireTokenByCode()` â†’ stores `id_token_claims` in the session cookie â†’ `/api/me` returns the session user or 401/403. The frontend discards all MSAL browser imports and replaces `AuthProvider` with a simple `fetch('/api/me')` pattern identical to xmcp's `AuthContext.tsx`.

The three most important version facts discovered: `@azure/msal-node` is at **v2.x** (NOT v5 â€” msal-node and msal-browser have independent version lines; the current npm release is v2.x for the node package); `iron-session` is at **v8.x** and fully supports Next.js 15 App Router `cookies()` API; the AWS SDK v3 `@aws-sdk/client-secrets-manager` uses a standard credential chain for on-prem (env vars â†’ `~/.aws/credentials` â†’ process credential provider) with no EC2 metadata endpoint required.

**Primary recommendation:** `pnpm add @azure/msal-node iron-session @aws-sdk/client-secrets-manager` + `pnpm remove @azure/msal-browser @azure/msal-react @microsoft/teams-js jose mock-jwks`. Implement BFF auth in four route handlers. Teams (AUTH-03 / DELV-03) is deferred to v1.1 â€” web-only is the correct scope for this pivot.

| Instead of | Could Use | Tradeoff |
|------------|-----------|----------|

### Pattern 1: Session-Cookie Auth Guard (replaces _middleware.ts jose JWT validator)

```typescript
// src/app/api/_middleware.ts  (REPLACE)
// Source: xmcp auth.py @role_required + iron-session pattern
import { getIronSession } from 'iron-session'
import { cookies } from 'next/headers'
import { SESSION_OPTIONS, type SessionData } from '@/auth/session'

export const runtime = 'nodejs'   // MANDATORY â€” iron-session touches Node crypto

export type AuthResult =
  | { sub: string; email: string; roles: string[] }
  | { error: 'unauthorized' }
  | { error: 'session_expired' }  // replaces 'token_expired'
  | { error: 'forbidden'; upn: string }

const REQUIRED_ROLE = 'KbAssistant.User'   // matches App Role 'Value' field in Entra

export async function getRequestUser(request: Request): Promise<AuthResult> {
  // Dev/test permissive stub preserved: no session AND non-production â†’ local-dev
  const cookieStore = await cookies()
  if (process.env.NODE_ENV !== 'production' && !cookieStore.get(SESSION_OPTIONS.cookieName)) {
    return { sub: 'local-dev', email: 'local@dev', roles: [REQUIRED_ROLE] }
  }

  const session = await getIronSession<SessionData>(cookieStore, SESSION_OPTIONS)
  if (!session.user) return { error: 'unauthorized' }

  // Check session freshness (iron-session handles cookie maxAge, but we can
  // also check a session timestamp for belt-and-suspenders expiry)
  const roles: string[] = session.user.roles ?? []
  if (!roles.includes(REQUIRED_ROLE)) {
    return { error: 'forbidden', upn: session.user.email ?? '' }
  }

  return { sub: session.user.oid, email: session.user.email, roles }
}
```

### Pattern 3: Auth Route Handlers

```typescript
// src/app/api/login/route.ts  (NEW)
export const runtime = 'nodejs'
import { type NextRequest, NextResponse } from 'next/server'
import { cookies } from 'next/headers'
import { getIronSession } from 'iron-session'
import { getCca } from '@/auth/msalClient'
import { SESSION_OPTIONS, type SessionData } from '@/auth/session'
import { env } from '@/config/env'

export async function GET(request: NextRequest) {
  const cca = getCca()
  const cookieStore = await cookies()
  const session = await getIronSession<SessionData>(cookieStore, SESSION_OPTIONS)

  const authCodeUrlResponse = await cca.getAuthCodeUrl({
    scopes: ['openid', 'profile', 'email'],
    redirectUri: `${env().APP_BASE_URL}/api/auth/callback`,
    // msal-node generates state + nonce internally when not provided
    // responseMode: POST is safer but requires form handling at callback;
    // QUERY works fine since we validate state via msal-node's internal check
    responseMode: ResponseMode.QUERY,
  })

  // msal-node does NOT expose state externally; it validates state + nonce
  // internally via its internal cache on the same CCA instance.
  // For multi-instance deployments, use SerializableTokenCache + distributed cache.
  // For single-instance (pilot on one Windows box), module-level CCA handles it.

  return NextResponse.redirect(authCodeUrlResponse)
}
```

```typescript
// src/app/api/auth/callback/route.ts  (NEW)
export const runtime = 'nodejs'
import { type NextRequest, NextResponse } from 'next/server'
import { cookies } from 'next/headers'
import { getIronSession } from 'iron-session'
import { getCca } from '@/auth/msalClient'
import { SESSION_OPTIONS, type SessionData } from '@/auth/session'
import { env } from '@/config/env'

export async function GET(request: NextRequest) {
  const { searchParams } = request.nextUrl
  const code = searchParams.get('code')
  const error = searchParams.get('error')

  if (error || !code) {
    // Entra returned an error (e.g. user declined consent) â€” redirect to /
    // so the app can re-initiate login
    return NextResponse.redirect(new URL('/', env().APP_BASE_URL))
  }

  const cca = getCca()
  let result
  try {
    result = await cca.acquireTokenByCode({
      code,
      scopes: ['openid', 'profile', 'email'],
      redirectUri: `${env().APP_BASE_URL}/api/auth/callback`,
    })
  } catch (err) {
    // CSRF / state mismatch or stale code â†’ restart login
    return NextResponse.redirect(new URL('/api/login', env().APP_BASE_URL))
  }

  const claims = result.idTokenClaims as Record<string, unknown>
  const cookieStore = await cookies()
  const session = await getIronSession<SessionData>(cookieStore, SESSION_OPTIONS)

  // xmcp pattern: session["user"] = result.get("id_token_claims")
  session.user = {
    oid: typeof claims.oid === 'string' ? claims.oid : '',
    email: typeof claims.preferred_username === 'string' ? claims.preferred_username : '',
    name: typeof claims.name === 'string' ? claims.name : '',
    roles: Array.isArray(claims.roles) ? (claims.roles as string[]) : [],
  }
  await session.save()

  return NextResponse.redirect(new URL('/', env().APP_BASE_URL))
}
```

```typescript
// src/app/api/me/route.ts  (NEW)

export const runtime = 'nodejs'
import { cookies } from 'next/headers'
import { getIronSession } from 'iron-session'
import { NextResponse } from 'next/server'
import { SESSION_OPTIONS, type SessionData } from '@/auth/session'

const REQUIRED_ROLE = 'KbAssistant.User'

export async function GET() {
  const session = await getIronSession<SessionData>(await cookies(), SESSION_OPTIONS)
  if (!session.user) {
    return NextResponse.json({ error: 'authentication_required' }, { status: 401 })

  if (!roles.includes(REQUIRED_ROLE)) {
    return NextResponse.json(
      { error: 'forbidden', upn: session.user.email },
      { status: 403 },
    )
  }
  return NextResponse.json({
    displayName: session.user.name,
    email: session.user.email,
    oid: session.user.oid,
    roles,
  })
}
```

```typescript
// src/auth/session.ts  (NEW)
import type { SessionOptions } from 'iron-session'
import { env } from '@/config/env'

export interface SessionData {
  user?: {
    oid: string
    email: string
    name: string
    roles: string[]
  }
}

// SESSION_OPTIONS reads SESSION_SECRET from env at call time (NOT module load)
// to ensure env() is populated. Cookie attributes follow xmcp pattern and
// Next.js 15 recommendations: HttpOnly + Secure + SameSite=Lax.
export function getSessionOptions(): SessionOptions {
  return {
    password: env().SESSION_SECRET,
    cookieName: 'kb_session',
    cookieOptions: {
      httpOnly: true,
      secure: process.env.NODE_ENV === 'production',
      sameSite: 'lax',
      path: '/',
      maxAge: 60 * 60 * 8, // 8-hour session (business-day scope)
    },
  }
}
// Alias for ergonomics
export const SESSION_OPTIONS = {
  get password() { return '' }, // placeholder â€” see Pitfall 2 note below
  cookieName: 'kb_session',
}
// Note: In practice, pass getSessionOptions() to getIronSession() calls rather
// than SESSION_OPTIONS directly. See Pitfall 2.
```

### Pattern 4: BFF AuthProvider (replaces MsalProvider)

Directly translated from xmcp `frontend/src/contexts/AuthContext.tsx`:

```typescript
// src/chat-ui/AuthProvider.tsx  (NEW â€” replaces src/app/providers.tsx MSAL variant)
'use client'
import { createContext, useContext, useEffect, useState, type ReactNode } from 'react'

export type AuthStatus = 'loading' | 'authenticated' | 'unauthenticated' | 'forbidden' | 'error'

interface User { displayName: string; email: string; oid: string; roles: string[] }

interface AuthState {
  status: AuthStatus
  user: User | null
  upn: string | null  // populated on 403
}

const AuthContext = createContext<AuthState | null>(null)

export function AuthProvider({ children }: { children: ReactNode }) {
  const [state, setState] = useState<AuthState>({ status: 'loading', user: null, upn: null })

  useEffect(() => {
    fetch('/api/me', { credentials: 'include' })
      .then(async (res) => {
        if (res.status === 401) {
          setState({ status: 'unauthenticated', user: null, upn: null })
        } else if (res.status === 403) {
          let upn = ''
          try { const body = await res.json(); upn = body.upn ?? '' } catch { /**/ }
          setState({ status: 'forbidden', user: null, upn })
        } else if (res.ok) {
          const user = await res.json() as User
          setState({ status: 'authenticated', user, upn: null })
        } else {
          setState({ status: 'error', user: null, upn: null })
        }
      })
      .catch(() => setState({ status: 'error', user: null, upn: null }))
  }, [])

  return <AuthContext.Provider value={state}>{children}</AuthContext.Provider>
}

export function useAuth(): AuthState {
  const ctx = useContext(AuthContext)
  if (!ctx) throw new Error('useAuth must be used within AuthProvider')
  return ctx
}
```

### Anti-Patterns to Avoid

- **Do not implement auth in Next.js Middleware (`middleware.ts`).** CVE-2025-29927 demonstrates that middleware-level auth in Next.js can be bypassed via `x-middleware-subrequest` header in some versions. The BFF route-handler + session-cookie approach sidesteps this class of vulnerability entirely.
- **Do not use edge runtime for auth routes.** `export const runtime = 'nodejs'` is MANDATORY on all four auth route handlers. iron-session uses Node.js crypto (`crypto.createCipheriv`) which is not available in the Edge Runtime.
- **Do not store raw id_token in the session cookie.** Store only `id_token_claims` (the decoded, server-verified claims object from `result.idTokenClaims`), matching xmcp's pattern exactly.
- **Do not attach Bearer tokens in the browser.** After pivot, `useChatStream` makes plain `fetch('/api/chat', { credentials: 'include' })` â€” the session cookie is sent automatically by the browser.

## Hot-Question Answers

### HQ-1: Next.js Implementation Path (Option a vs b)

**Recommendation: Option (a) â€” keep Next.js, add auth in route handlers.**

Reasoning:
1. xmcp's pattern is "backend owns both auth and app serving" â€” Next.js App Router route handlers ARE the backend for this app. There is no meaningful difference in shape from Flask routes.
2. `@azure/msal-node` is the direct Node.js equivalent of xmcp's `msal.ConfidentialClientApplication`. The API surface maps 1:1: `getAuthCodeUrl()` = `initiate_auth_code_flow()`, `acquireTokenByCode()` = `acquire_token_by_auth_code_flow()`.
3. Option (b) (separate Express/Fastify BFF) adds a second process to manage, requiring NSSM to supervise two processes or a process-manager layer. This makes on-prem Windows deployment more complex with no benefit for a pilot-scale app.
4. `iron-session` v8 supports Next.js 15 App Router `cookies()` API natively â€” session management is 3 lines of code.

**msal-node API surface confirmed (HIGH confidence, Microsoft Learn docs):**
- `ConfidentialClientApplication(config)` â€” constructor, takes `auth.clientId`, `auth.authority`, `auth.clientSecret`
- `getAuthCodeUrl(request)` â†’ `Promise<string>` â€” generates the Entra login URL. Pass `scopes`, `redirectUri`, optionally `responseMode`
- `acquireTokenByCode(request)` â†’ `Promise<AuthenticationResult>` â€” exchanges code for tokens. Pass `code`, `redirectUri`, `scopes`. Returns `result.idTokenClaims` with the `roles` array.
- `acquireTokenSilent(request)` â€” not needed for this app (session cookie handles persistence; we only need the id_token_claims at login time)
- **PKCE state management:** msal-node manages PKCE and state internally on the CCA instance. For a single-instance pilot app, a module-level singleton CCA handles this correctly. For multi-instance deployment, use `SerializableTokenCache` with distributed storage (Redis/DB) â€” flag as OPEN QUESTION for user.

**iron-session v8 API (HIGH confidence, README verified):**
```typescript
import { getIronSession } from 'iron-session'
import { cookies } from 'next/headers'

// In a route handler:
const session = await getIronSession<SessionData>(await cookies(), {
  password: process.env.SESSION_SECRET!, // min 32 chars
  cookieName: 'kb_session',
  cookieOptions: { httpOnly: true, secure: true, sameSite: 'lax', path: '/', maxAge: 60*60*8 }
})
session.user = { oid, email, name, roles }
await session.save()
```
- `session.destroy()` â€” clears cookie (use for `/api/logout`)
- Cookie defaults already include `httpOnly: true`, `secure: true` (can override for dev), `sameSite: 'lax'`, `path: '/'`
- The `lax` same-site setting is correct for the auth code flow redirect: the Entra redirect back to `/api/auth/callback` is a cross-site redirect, and `SameSite=Lax` allows cookies to be sent on top-level navigations (GET redirects), which is exactly what the callback is.

### HQ-2: App Role Setup and id_token_claims.roles Verification

**App Role configuration (HIGH confidence, Microsoft Learn official docs):**

1. In Entra Admin Center â†’ App registrations â†’ your app â†’ **App roles** â†’ Create app role:
   - Display name: `KB Assistant User`
   - Value: `KbAssistant.User` (this is the string that appears in `roles` claim; choose this name deliberately)
   - Allowed member types: `Users/Groups`
   - Description: `Access to KB Assistant pilot`
2. In **Enterprise applications** â†’ your app â†’ **Users and groups** â†’ Add user/group â†’ assign the pilot security group to `KbAssistant.User`

**Is `roles` ALWAYS in id_token when user has the role?** YES â€” from Microsoft docs: "For an app that signs in users, the roles claims are included in the ID token." When a user who has been assigned the App Role authenticates with auth code flow (which returns an id_token), the `roles` array will contain the role value. No extra scope is needed â€” `openid profile email` is sufficient to get id_token with roles.

**Does admin consent need to be granted separately?** The docs note admin consent is needed for *application permissions* (app-to-app). For user sign-in scenarios with App Roles assigned to users/groups, admin consent is NOT required separately â€” the role assignment in Enterprise App is the authorization mechanism. However, the pilot team should confirm with the MMC IT admin whether the tenant requires admin pre-consent on first login; if `"assignment required": true` is set on the Enterprise App, only assigned users can sign in at all.

**Exact guard pattern (from xmcp auth.py):**
```typescript
const REQUIRED_ROLE = 'KbAssistant.User'
const roles: string[] = (claims.roles as string[]) ?? []
if (!roles.includes(REQUIRED_ROLE)) {
  return { error: 'forbidden', upn: claims.preferred_username as string }
}
```

**What if roles is absent from id_token?** This happens only if: (a) the user was never assigned the role in Enterprise App, or (b) the App Role was never created in the App Registration. In both cases `roles` key is absent from claims â€” treat `undefined` as an empty array (same as xmcp: `user.get("roles", [])`).

### HQ-3: AWS Secrets Manager Access from On-Prem Windows Node.js

**Credential chain (MEDIUM confidence, AWS SDK v3 docs verified):**

The AWS SDK v3 for JavaScript tries credential providers in this order on a non-EC2 host:
1. `fromEnv()` â€” `AWS_ACCESS_KEY_ID` + `AWS_SECRET_ACCESS_KEY` + optional `AWS_SESSION_TOKEN` from environment variables
2. `fromSSO()` â€” AWS IAM Identity Center
3. `fromIni()` â€” `%USERPROFILE%\.aws\credentials` (Windows) or `~/.aws/credentials`
4. `fromProcess()` â€” process credential provider (credential_process in aws config)
5. (EC2 metadata endpoint is tried last but will time out on non-EC2)

**Recommended on-prem Windows pattern (matching xmcp's boto3 approach exactly):**

```typescript
// src/config/secrets.ts  (NEW)
// Source: xmcp chat_app/secrets.py load_secrets()
import { env } from '@/config/env'

const SECRET_NAME = process.env.AWS_SECRET_NAME ?? '/mmc/cts/kb-assistant'
const AWS_REGION = process.env.AWS_REGION ?? 'us-east-1'

let _cachedSecrets: Record<string, string> | null = null

export async function loadSecrets(): Promise<Record<string, string>> {
  if (_cachedSecrets) return _cachedSecrets

  // Try AWS Secrets Manager first (production)
  try {
    const { SecretsManagerClient, GetSecretValueCommand } = await import('@aws-sdk/client-secrets-manager')
    const client = new SecretsManagerClient({ region: AWS_REGION })
    const response = await client.send(new GetSecretValueCommand({ SecretId: SECRET_NAME }))
    const secrets = JSON.parse(response.SecretString ?? '{}') as Record<string, string>
    _cachedSecrets = secrets
    return secrets
  } catch (err) {
    // AWS Secrets Manager unavailable or credentials not configured â†’ fall back to env
    console.info('[secrets] AWS Secrets Manager unavailable, falling back to env:', err)
  }

  // Fallback: already loaded via process.env / .env.local
  _cachedSecrets = {}
  return _cachedSecrets
}
```

**On-prem credential setup options (user must choose one at deploy time):**
- **Option A (simplest):** IAM user access keys stored in `%USERPROFILE%\.aws\credentials` on the Windows box under the NSSM service account's profile, or set as system environment variables `AWS_ACCESS_KEY_ID` + `AWS_SECRET_ACCESS_KEY` (same env var pattern xmcp uses via `$env:...`).
- **Option B (MMC-preferred if available):** IAM Roles Anywhere â€” allows on-prem hosts with X.509 certs to assume IAM roles without static access keys. More complex setup; check with MMC IT whether this is already configured for Atlas.
- **Option C:** AWS IAM Identity Center (SSO) via `fromSSO()` â€” requires SSO portal login; less practical for a service account.

### HQ-5: Session Cookie Settings + CSRF

**Cookie attributes for `/api/auth/callback` Set-Cookie:**

| Attribute | Value | Reason |
|-----------|-------|--------|
| `HttpOnly` | true | Prevents JS access â€” xmcp's Flask session default |
| `Secure` | true (prod) / false (dev) | HTTPS only in production |
| `SameSite` | `lax` | Allows Entra auth code redirect callback (top-level GET navigation); blocks CSRF for POST requests |
| `Path` | `/` | Full-app scope |
| `maxAge` | 8 hours (28800s) | Business-day session, explicit timeout |

**Does `SameSite=Lax` protect `/api/chat` POST from CSRF?** YES for the pilot. `SameSite=Lax` means the cookie is sent on top-level navigations (GET) but NOT on cross-site `fetch()` or `XMLHttpRequest` with `POST`. Since the KB assistant is an MMC-internal app accessed only from MMC network, an attacker would need to be inside the MMC network to make a cross-site request. For extra security, a `csrf_token` double-submit pattern can be added in v1.1 â€” not required for pilot. xmcp does not implement CSRF protection on its JSON API endpoints either.

**`__Host-` cookie prefix:** Desirable security hardening but NOT compatible with IIS reverse proxy without additional configuration (the `__Host-` prefix requires `Secure`, `Path=/`, and no `Domain` attribute â€” compatible, but if IIS strips `Set-Cookie` headers unexpectedly it fails silently). Use the plain `kb_session` name for pilot; add `__Host-` prefix in v1.1 after verifying the IIS header pass-through.

**iron-session handles SameSite=Lax automatically.** Default iron-session cookie options already set `sameSite: 'lax'`, `httpOnly: true`, `secure: true`, `path: '/'`. These are correct and do NOT need to be overridden.

## Pitfalls

### Pitfall 1: `export const runtime = 'nodejs'` is MANDATORY on all auth route handlers
**What goes wrong:** If omitted, Next.js may attempt to run the route handler on the Edge Runtime. iron-session uses Node.js `crypto.createCipheriv` which throws `crypto is not defined` or similar on Edge. `@azure/msal-node` also uses Node.js `crypto` and `fs` modules.
**Prevention:** Add `export const runtime = 'nodejs'` as the first line of every file in `src/app/api/login/`, `src/app/api/auth/callback/`, `src/app/api/logout/`, `src/app/api/me/`, and the new `src/app/api/_middleware.ts`. The existing `src/app/api/chat/route.ts` already has this.

### Pitfall 2: iron-session `SESSION_SECRET` must be â‰Ą32 characters and must be read at runtime not module-load time
**What goes wrong:** If `SESSION_OPTIONS` is a top-level `const` that reads `env().SESSION_SECRET` at module load, and the secret is loaded from AWS Secrets Manager asynchronously, the module may initialize before secrets are available.
**Prevention:** Pass a function or lazily-evaluated options object to `getIronSession()`. The recommended pattern in the Architecture section uses `getSessionOptions()` (called at request time) rather than a top-level constant.

### Pitfall 3: msal-node PKCE state is in-process only
**What goes wrong:** If the Node.js process restarts between `/api/login` (which generates PKCE verifier in the CCA internal cache) and `/api/auth/callback` (which validates state against that cache), the callback will throw `invalid_state` or similar MSAL error.
**Prevention:** For single-instance pilot: acceptable. For multi-instance: requires `SerializableTokenCache` with distributed backing store. Confirm with user (Open Question 4).

### Pitfall 4: `/api/auth/callback` redirect URI MUST match exactly in both Entra App Registration and msal-node request
**What goes wrong:** Even a trailing slash difference (`/api/auth/callback` vs `/api/auth/callback/`) causes Entra to reject the callback with `AADSTS50011: The redirect URI specified in the request does not match the redirect URIs configured for the application`.
**Prevention:** Use a single canonical `APP_BASE_URL` env var. Build the redirect URI as `${env().APP_BASE_URL}/api/auth/callback` in both `getAuthCodeUrl()` and `acquireTokenByCode()`. Verify the Entra App Registration redirect URI is exactly `https://<host>/api/auth/callback` (no trailing slash). Platform type must be `Web` NOT `Single-page application`.

### Pitfall 5: `roles` claim is absent from id_token if App Role is not created + user not assigned
**What goes wrong:** `claims.roles` is `undefined` (not an empty array). Code that does `claims.roles.includes(...)` will throw.
**Prevention:** Always access as `(claims.roles as string[] | undefined) ?? []`. This is the xmcp pattern: `user.get("roles", [])`.

### Pitfall 6: IIS ARR buffers SSE streams unless `responseBufferLimit="0"` is set
**What goes wrong:** `/api/chat` returns SSE tokens one at a time, but IIS ARR buffers the response until it closes. The user sees a loading spinner for the full LLM response time, then all tokens appear at once.
**Prevention:** In ARR server proxy settings, set `responseBufferLimit="0"`. Add `X-Accel-Buffering: no` response header. This is documented in xmcp `DEPLOYMENT.md` ("SSE Streaming Issues" section) â€” the same fix applies.

### Pitfall 7: `SameSite=Lax` + IIS reverse proxy â€” cookies not sent on first redirect
**What goes wrong:** Entra redirects back to `/api/auth/callback`. If the iron-session pre-auth cookie (state storage for PKCE) was set with `SameSite=Strict`, it won't be sent on this cross-site redirect, and the callback will fail to find the PKCE state.
**Prevention:** Use `SameSite=Lax` (not Strict). Lax allows cookies on top-level GET navigations (cross-site redirects). The iron-session default is already `lax`.

### Pitfall 8: Playwright E2E fixture cannot seal a real iron-session cookie without `SESSION_SECRET`
**What goes wrong:** Naively trying to bake a pre-sealed iron-session cookie in the test fixture requires the same AES key used in production â€” sharing that key in test code is a security anti-pattern.
**Prevention:** Mock `/api/me` at the network layer via `page.route('/api/me', ...)` instead. This completely bypasses the session layer and returns a synthetic authenticated response. See `mockSession.ts` pattern in HQ-8.

### Pitfall 9: `NEXT_PUBLIC_ENTRA_CLIENT_ID` baked at build time â€” post-pivot may need removal
**What goes wrong:** Next.js bakes `NEXT_PUBLIC_*` vars into the JavaScript bundle at `pnpm build`. If `NEXT_PUBLIC_ENTRA_CLIENT_ID` is still in the bundle (even unused), it leaks the client ID to every browser. Post-pivot, no client-side code reads this var â€” it's dead code.
**Prevention:** Remove `NEXT_PUBLIC_ENTRA_*` from `env.ts` and from `next.config.ts` env injection after verifying no client component references them. The client ID IS NOT a secret (it's in every Entra OAuth URL anyway) but leaking it unnecessarily is messy.

### Pitfall 10: Next.js 15 `cookies()` is async â€” requires `await`
**What goes wrong:** In Next.js 14, `cookies()` was synchronous. In Next.js 15 (this project: 15.5.6), it is async. Code like `const c = cookies()` without `await` returns a Promise, not the CookieStore. Calling `c.get(...)` returns `undefined` silently.
**Prevention:** Always `const cookieStore = await cookies()` before passing to `getIronSession`. This is shown in all examples above. The Next.js 15 docs confirm this change.

### Pitfall 11: msal-node `ConfidentialClientApplication` must NOT be re-created per request
**What goes wrong:** Creating a new `ConfidentialClientApplication` on every request defeats the internal token cache and PKCE state tracking, and adds ~50-100ms latency per login request (MSAL fetches OIDC metadata on init).
**Prevention:** Module-level singleton, exactly as xmcp does it (`_build_msal_app()` is called per-auth-route but the flask session shares the cache). For Next.js, a module-level `let _cca` with lazy init (see `msalClient.ts` pattern above) is correct.

### Pitfall 12: AWS Secrets Manager adds ~200-500ms latency on cold start
**What goes wrong:** If `loadSecrets()` is called synchronously during module init (e.g., at the top of `env.ts`), the first request after a cold start will be slow, and if Secrets Manager is unreachable the app will fail to start.
**Prevention:** Load secrets asynchronously during server startup (in `instrumentation.ts` if Next.js instrumentation is available, or lazily on first request with caching). The `_cachedSecrets` module-level cache means subsequent requests are instant. Ensure the fallback path (`.env.local` for dev) works independently.
