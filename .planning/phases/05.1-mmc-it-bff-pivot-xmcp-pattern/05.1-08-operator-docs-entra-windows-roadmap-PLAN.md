# Entra App Registration Setup — KB Assistant

This document is the one-time manual setup the KB Assistant owner performs in the MMC Entra tenant admin portal AFTER the code is pushed to the repo and the on-prem Windows deploy is ready (see `docs/deploy-windows.md`).

**Time required:** ~20 minutes.
**Prerequisites:** Entra admin role in the MMC tenant (Application Administrator OR Cloud Application Administrator OR Global Administrator). If you don't have this, file a ticket with MMC IT Identity team.

**What you're creating:** a new App Registration named `KB Assistant` with:
- Platform: `Web` (NOT Single-page application — the old Phase-5 SPA registration is superseded)
- Redirect URI: `https://<app-host>/api/auth/callback` (exact match — see Pitfall 4 below)
- Client secret (stored in AWS Secrets Manager, not in source control)
- App Role `KbAssistant.User` (access gate — see Pitfall 5 below)
- Pilot users/group assignment to the App Role in Enterprise Applications

---

## Step 1 — Create the App Registration

1. Sign in to https://entra.microsoft.com with your Entra admin account.
2. Navigate: **Identity → Applications → App registrations → New registration**.
3. Fill in the form:
   - **Name:** `KB Assistant`
   - **Supported account types:** `Accounts in this organizational directory only (Single tenant)` — if MMC is the only tenant; choose differently per your tenant topology.
   - **Redirect URI:**
     - Platform: `Web`
     - URI: `https://<app-host>/api/auth/callback`
     - Replace `<app-host>` with your Windows Server FQDN (e.g. `usdf11v1784.mercer.com`). This value MUST match `APP_BASE_URL` in the deployed env exactly — no trailing slash, no port suffix, no path variation. **Pitfall 4** — even a single-character mismatch causes `AADSTS50011` at login.
4. Click **Register**.
5. On the Overview page, copy the **Application (client) ID** and **Directory (tenant) ID**. Save both — they become `ENTRA_CLIENT_ID` and `ENTRA_TENANT_ID` in AWS Secrets Manager (see Step 6).

**How you know it worked:** the Overview page shows the app name, client ID, tenant ID, and the redirect URI under "Redirect URIs".

---

## Step 2 — Create a Client Secret

1. Still on the App Registration page, click **Certificates & secrets** in the left nav.
2. Click **Client secrets → New client secret**.
3. Description: `kb-assistant-bff-secret` (or any name — this is operator-visible only).
4. Expires: `24 months` (or per MMC policy — set a rotation reminder).
5. Click **Add**.
6. **IMPORTANT:** Copy the `Value` column IMMEDIATELY. This is the only time Entra will show it — if you close the page without copying, you must delete and create a new secret. Save this value as `ENTRA_CLIENT_SECRET` in your notes (it's written to AWS Secrets Manager in Step 6).

**How you know it worked:** the secret appears in the list with an expiry date. Treat the copied secret value like a password — anyone with it can impersonate the app.

---

## Step 3 — Define the App Role

1. Click **App roles** in the left nav.
2. Click **Create app role**.
3. Fill in:
   - **Display name:** `KB Assistant User`
   - **Allowed member types:** `Users/Groups`
   - **Value:** `KbAssistant.User` ← This exact string must match the `REQUIRED_ROLE` constant in `src/app/api/_middleware.ts`. Do not change casing or add spaces.
   - **Description:** `Access to KB Assistant pilot`
4. Click **Apply**.

**How you know it worked:** the App Role appears in the list. When a user with this role signs in, `id_token.roles` will be `["KbAssistant.User"]`. When a user without the role signs in, `id_token.roles` is undefined (absent) — `_middleware.ts` coerces to `[]` per **Pitfall 5** and returns `forbidden`.

---

## Step 4 — Grant API Permissions

1. Click **API permissions** in the left nav.
2. The default `Microsoft Graph → User.Read` is sufficient. If it's not there, click **Add a permission → Microsoft Graph → Delegated permissions → User.Read → Add permissions**.
3. The default delegated permissions `openid`, `profile`, `email` are already granted for any Entra auth app — confirm they appear. If not, **Add a permission → Microsoft Graph → Delegated permissions** → check `openid`, `profile`, `email`.
4. Click **Grant admin consent for <MMC tenant name>**. Confirm. All listed permissions should show "Granted for <tenant>" status with a green check.

**How you know it worked:** all four permissions (`openid`, `profile`, `email`, `User.Read`) show "Granted" status with green checkmarks.

---

## Step 5 — Assign Pilot Users/Group to the App Role

This is the step that actually controls who can sign in. Without it, even users in the correct tenant are blocked.

1. Navigate to **Identity → Applications → Enterprise applications**.
2. Find `KB Assistant` (search by name). The Enterprise Application was auto-created when you registered the App — this is the "user-facing" side of the registration.
3. Click **Users and groups → Add user/group**.
4. Under "Users and groups": select the pilot security group (or individual users). Ask MMC IT Identity for the appropriate pilot group name, or create a new one.
5. Under "Select a role": select `KB Assistant User`.
6. Click **Assign**.

**Optional hardening:** navigate to **Properties** of the same Enterprise Application and set **Assignment required?** to `Yes`. This means ONLY users in the assigned group can sign in at all (non-assigned MMC users get blocked at the Entra sign-in screen before they even reach the app). Recommended for the pilot.

**How you know it worked:** the pilot group appears in the Users and groups list with the `KB Assistant User` role next to it.

---

## Step 6 — Write Secrets to AWS Secrets Manager

Three values from Steps 1–2 go to AWS Secrets Manager:

- `ENTRA_CLIENT_ID` (Step 1)
- `ENTRA_TENANT_ID` (Step 1)
- `ENTRA_CLIENT_SECRET` (Step 2)

You also need to generate these values:
- `SESSION_SECRET` — 64-character random string (iron-session AES key). Generate with `openssl rand -base64 48` or equivalent.
- `LLM_API_KEY` — MGTI API key (obtain from the MGTI team if not already on the box).
- `LLM_BASE_URL` — MGTI endpoint URL.

Create the secret via the AWS CLI on the Windows box OR on your laptop if your IAM user has `secretsmanager:CreateSecret` permission:

```powershell
$body = @{
  SESSION_SECRET = '<64-char-random>'
  ENTRA_CLIENT_ID = '<client-id>'
  ENTRA_TENANT_ID = '<tenant-id>'
  ENTRA_CLIENT_SECRET = '<client-secret>'
  LLM_API_KEY = '<mgti-api-key>'
  LLM_BASE_URL = '<mgti-endpoint>'
} | ConvertTo-Json

aws secretsmanager create-secret `
  --name /mmc/cts/kb-assistant `
  --region us-east-1 `
  --secret-string $body
```

If the secret already exists (re-running this doc), use `update-secret` instead of `create-secret`.

**How you know it worked:** `aws secretsmanager get-secret-value --secret-id /mmc/cts/kb-assistant --region us-east-1` returns a `SecretString` containing your JSON. The app's `loadSecrets()` will pick these up at startup.

## Step 7 — Smoke Test the Sign-in Flow

Once `docs/deploy-windows.md` Steps 1-6 are also done AND the app is deployed:

1. In a clean browser window (or Incognito), visit `https://<app-host>/`.
2. You should be redirected to `login.microsoftonline.com/<tenant>/oauth2/v2.0/authorize?client_id=<your-client-id>&...`.
3. Sign in with a user who is in the pilot assignment group.
4. Consent to the requested scopes (first sign-in only).
5. You should land back on `https://<app-host>/` with the role-select screen visible.
6. Open DevTools → Application → Cookies. Confirm a cookie named `kb_session` is present, `HttpOnly`, `Secure`, `SameSite=Lax`.

**If you see `AADSTS50011`:** the redirect URI in Step 1 doesn't match `APP_BASE_URL` + `/api/auth/callback` exactly. Fix by editing the App Registration's Redirect URIs.

**If you see a 403 / "access restricted" page:** the user is signed in but missing the `KbAssistant.User` App Role. Verify Step 5 assignment OR check the user's Enterprise Application assignment.

**If you see `AADSTS7000218`:** the client secret is wrong or expired. Regenerate in Step 2 and update AWS Secrets Manager (Step 6).

---

## Rotation / maintenance

- **Client secret expires:** set a calendar reminder 60 days before the Step 2 expiry date. Rotate by creating a new secret, updating AWS Secrets Manager, restarting the Scheduled Task, and only then deleting the old secret.
- **Pilot user additions:** re-run Step 5 for each new user/group. No app restart needed.
- **Moving from pilot to v1.1 (Teams, additional hosts):** add additional redirect URIs under App Registration → Authentication. Do NOT delete the existing `https://<app-host>/api/auth/callback` URI.

---

```markdown
# Windows Server Deployment — KB Assistant

One-time setup for the on-prem Windows Server box that hosts the KB Assistant. After this runbook, the GitHub Actions `deploy` workflow (`.github/workflows/deploy.yml`) will push updates automatically on every merge to `main`.

**Time required:** ~90 minutes for the first deploy (most of it waiting for installs).
**Prerequisites:**
- Windows Server with Administrator (or equivalent) access.
- Domain-joined box inside the MMC corporate network.
- AWS credentials already configured on the box (per planning context — credentials at `%USERPROFILE%\.aws\credentials` or machine env vars).
- Entra App Registration complete (see `docs/entra-app-registration-setup.md`).
- AWS Secrets Manager secret `/mmc/cts/kb-assistant` exists (see entra-app-registration-setup.md Step 6).

---

## Step 1 — Install Node.js 20

1. Download the Node.js 20.x LTS Windows Installer (`.msi`) from https://nodejs.org/en/download.
2. Run the installer. Default location is `C:\Program Files\nodejs\`. Accept all defaults.
3. Verify in a NEW PowerShell window (open after install so PATH refreshes):
   ```powershell
   node --version   # → v20.x.x
   npm --version    # → 10.x.x
   ```

**Pitfall:** Installing Node.js under an individual user profile instead of system-wide means the Scheduled Task running as a service account won't find `node.exe`. Always install system-wide.

---

## Step 2 — Verify AWS Credentials

The app's `loadSecrets()` reads `/mmc/cts/kb-assistant` from AWS Secrets Manager (us-east-1) via the standard AWS SDK credential chain: env vars → `%USERPROFILE%\.aws\credentials` → process credential provider.

1. Open PowerShell as the account the Scheduled Task will run as (usually `LocalSystem` or a service account — per your tenant policy).
2. Test:
   ```powershell
   aws secretsmanager get-secret-value --secret-id /mmc/cts/kb-assistant --region us-east-1
   ```
3. Expected: JSON response with a `SecretString` field containing your secret JSON. If you get `AccessDenied` or `NoCredentialProviders`, credentials are not reachable from this account. Check:
   - Env vars `AWS_ACCESS_KEY_ID` + `AWS_SECRET_ACCESS_KEY` are set at machine scope (`[System.Environment]::GetEnvironmentVariable('AWS_ACCESS_KEY_ID', 'Machine')`).
   - OR the service account's `%USERPROFILE%\.aws\credentials` has a `[default]` profile.

Do NOT proceed to Step 3 until this call succeeds under the account the Scheduled Task will run as.

---

## Step 3 — Create the Deploy Directory

Pick the deploy root. Default per the `deploy.yml` workflow is `D:\kbroles\.next\standalone`. You can override this via a GitHub Repository Variable `KBASSISTANT_DEPLOY_ROOT` — see Step 6.

```powershell
New-Item -ItemType Directory -Path 'D:\kbroles\.next\standalone' -Force
New-Item -ItemType Directory -Path 'D:\logs' -Force
```

The standalone bundle will be extracted into this path by each deploy. Initially it's empty — the first workflow run populates it.

---

## Step 4 — Create the Windows Scheduled Task

The Node.js server runs as a Scheduled Task (NOT NSSM — MMC standard per this deploy). The task triggers at system startup and restarts automatically on failure.

1. Open **Task Scheduler** (`taskschd.msc`).
2. Click **Create Task…** (NOT Create Basic Task — we need the advanced settings).
3. **General tab:**
   - Name: `KbAssistant`
   - Description: `KB Knowledge Assistant — Next.js standalone server`
   - Security options → **Run whether user is logged on or not** (checked).
   - User account: the service account used in Step 2 (`LocalSystem` or named service account).
   - Configure for: Windows Server (your version).
4. **Triggers tab:**
   - New → Begin the task: `At startup`.
   - (Optional) add a second trigger `Daily` for belt-and-suspenders — if the box runs for weeks without reboot, a daily restart keeps memory use bounded. Skip if not desired.
5. **Actions tab:**
   - New → Action: `Start a program`.
   - Program/script: `C:\Program Files\nodejs\node.exe`
   - Add arguments: `D:\kbroles\.next\standalone\server.js`
   - Start in: `D:\kbroles\.next\standalone`
6. **Conditions tab:** uncheck **Start the task only if the computer is on AC power** (may be relevant for VMs — doesn't matter on a server but tighten to match your policy).
7. **Settings tab:**
   - **Allow task to be run on demand**: CHECKED (required — the GitHub Actions workflow uses `schtasks /run` which is an on-demand start).
   - **If the task fails, restart every:** `1 minute`, **Attempt to restart up to:** `3 times`.
   - **If the running task does not end when requested, force it to stop:** CHECKED.
   - **If the task is already running, then the following rule applies:** `Do not start a new instance`.
   - Leave "Stop the task if it runs longer than" UNCHECKED (the server is long-running by design).
8. Click **OK**. Provide the service account password if prompted.

**Environment variables:** the Scheduled Task inherits the SYSTEM or service-account's environment at task start. Set the following at MACHINE scope so both the runner and the task see them:

```powershell
[System.Environment]::SetEnvironmentVariable('NODE_ENV', 'production', 'Machine')
[System.Environment]::SetEnvironmentVariable('PORT', '3001', 'Machine')
[System.Environment]::SetEnvironmentVariable('HOSTNAME', '127.0.0.1', 'Machine')
[System.Environment]::SetEnvironmentVariable('APP_BASE_URL', 'https://<app-host>', 'Machine')
[System.Environment]::SetEnvironmentVariable('AWS_SECRET_NAME', '/mmc/cts/kb-assistant', 'Machine')
[System.Environment]::SetEnvironmentVariable('AWS_REGION', 'us-east-1', 'Machine')
```

Reboot the box OR restart the Scheduled Task after setting machine-scope env vars — existing processes won't see them otherwise.

**Logging:** by default Scheduled Task output is not captured. To capture stdout/stderr:
- **Simpler:** wrap the action in a PowerShell script that redirects output to a file. Create `D:\kbroles\start.ps1`:
  ```powershell
  & 'C:\Program Files\nodejs\node.exe' 'D:\kbroles\.next\standalone\server.js' *>&1 | Tee-Object -FilePath 'D:\logs\kbassistant.log' -Append
  ```
  Change the Action in Step 5 to `powershell.exe` with arguments `-File D:\kbroles\start.ps1`.
- **Rotation:** use a daily scheduled task to rotate `D:\logs\kbassistant.log` — or let it grow and monitor disk. Pilot-scale is fine either way.

**How you know it worked:** `schtasks /query /tn KbAssistant` returns the task. After an `/run`, `netstat -ano | findstr :3001` shows a listener. `Invoke-RestMethod http://localhost:3001/api/health` returns `{ "status": "ok" }` (assuming the deploy dir is populated — so run this AFTER Step 7).

## Step 5 — Configure IIS Reverse Proxy

IIS terminates TLS at 443 and forwards plain HTTP to `127.0.0.1:3001`. SSL cert comes from the MMC network team (PEM or PFX).

1. In Server Manager, add the **Web Server (IIS)** role if not already installed.
2. Download and install:
   - **URL Rewrite** module (Microsoft download).
   - **Application Request Routing** (ARR) module.
3. In IIS Manager:
   - Create a new website bound to 443 with the MMC-provided SSL cert.
   - Point the site root to an empty directory (e.g. `D:\inetpub\kbassistant`) — IIS just needs a physical path for the rewrite rule.
4. Enable ARR proxy: IIS Manager root → Application Request Routing Cache → Server Proxy Settings → **Enable proxy** (checked).
5. **CRITICAL for SSE:** In the same ARR settings, set **Response buffer threshold (KB)** to `0`. Without this, `/api/chat` SSE tokens buffer until the stream closes (Pitfall 6).
6. Add the following `web.config` to the site root (`D:\inetpub\kbassistant\web.config`):

```xml
<?xml version="1.0" encoding="UTF-8"?>
<configuration>
  <system.webServer>
    <rewrite>
      <rules>
        <rule name="ReverseProxyToNode" stopProcessing="true">
          <match url="(.*)" />
          <action type="Rewrite" url="http://127.0.0.1:3001/{R:1}" />
          <serverVariables>
            <set name="HTTP_X_FORWARDED_PROTO" value="https" />
          </serverVariables>
        </rule>
      </rules>
    </rewrite>
    <httpProtocol>
      <customHeaders>
        <add name="X-Accel-Buffering" value="no" />
      </customHeaders>
    </httpProtocol>
  </system.webServer>
</configuration>
```

`X-Forwarded-Proto: https` is important so iron-session's `secure: true` cookie attribute is consistent with the HTTPS front-end.
