### window.location spy pattern chosen

jsdom 24+ rejects `vi.spyOn(window.location, 'href', 'set')` because `Location.prototype.href` has `configurable: false` on the native object. I used a **Proxy-wrapped location** installed via `Object.defineProperty(window, 'location', { value: Proxy, configurable: true })` instead:

```ts
const hrefSpy = vi.fn()
const realLocation = window.location
Object.defineProperty(window, 'location', {
  configurable: true,
  enumerable: true,
  value: new Proxy(realLocation, {
    set(target, prop, value) {
      if (prop === 'href') { hrefSpy(value); return true }
      return Reflect.set(target, prop, value)
    },
    get(target, prop) { return Reflect.get(target, prop) },
  }),
})
// ... assertion ...
// finally: restore via another defineProperty with the real location
```

This is surgical (only `set href` is intercepted; all reads pass through) and restores cleanly in each test's `finally` block so later tests in the same file aren't affected.

## Decisions Made

1. **ChatPage skeleton condition extended to include `status === 'forbidden'`** — Rule 1 (Bug) auto-fix. Plan sample code gated the skeleton on `'loading'|'unauthenticated'|'error'` only. Without `'forbidden'` the render path would attempt to mount `<ChatSurface>` / `<RoleSelect>` during the tiny window before `router.replace('/access-denied')` fires, causing a potential flash of chat UI (and worse — an access-denied user briefly sees role selection). Added `'forbidden'` to the skeleton predicate for consistency with how unauthenticated is handled.

2. **`UseChatStreamOptions` retained as `Record<string, never>` rather than removed** — keeps the second-argument shape stable for callers. Removing the option parameter entirely would be a breaking signature change; keeping it as an empty record makes the `useChatStream(onEvent, {})` call pattern compile cleanly and leaves a documented home for future non-auth options.

3. **`handleRetry` changed from async to sync** — once the `await acquireToken(null)` step was removed, there was no remaining async work before `dispatch`. The `MessageList onRetry` wrapper was simplified from `(id) => { void handleRetry(id) }` to `handleRetry` directly.

4. **Added 3 unit tests to `useChatStream.test.tsx` covering pre-stream 401/403 branching** — Rule 2 (Missing Critical). Phase 5 had only integration-level coverage for these wire codes via `ChatSurface.test.tsx`. Plan 06 will continue to modify ChatSurface mocks; unit-level coverage of the hook itself is defense-in-depth so regressions don't hide behind ChatSurface mock changes.
