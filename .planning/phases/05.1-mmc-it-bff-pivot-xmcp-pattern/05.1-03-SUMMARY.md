## Accomplishments

- `/api/login` 307 redirects to the Entra authorize URL via `getCca().getAuthCodeUrl({scopes:['openid','profile','email'], redirectUri:`${APP_BASE_URL}/api/auth/callback`, responseMode: ResponseMode.QUERY})`. Pitfall-4 exact-match redirect URI construction; Pitfall-3 single-instance CCA reuses PKCE state across the pair.
- `/api/auth/callback` exchanges the code via `acquireTokenByCode`, reads `idTokenClaims`, coerces `roles` to `[]` when undefined (Pitfall 5), and persists the `{oid, email:preferred_username, name, roles}` subset via `saveSession()`. No raw JWT strings in the session cookie.
- Error paths covered: `?error=` → redirect / without saveSession; missing code → /api/login; acquireTokenByCode throw (stale code / state mismatch) → /api/login (Pitfall-3 recovery).
- `/api/logout` calls `clearSession()` and 307 redirects to /. Local-only — no Entra global logout (xmcp-matching behaviour).
- `/api/me` returns the BFF contract the frontend AuthProvider (Plan 05) will consume:
  - `200 {displayName, email, oid, roles}` when the session user has the `KbAssistant.User` role
  - `401 {error:'authentication_required'}` when no session
  - `403 {error:'forbidden', upn:<email>}` when authenticated but lacking the required role
  - `Cache-Control: no-store` on every branch
- All four handlers declare `export const runtime = 'nodejs'` as the first line after imports (Pitfall 1) and call `await loadSecrets()` at the top of their GET function (Pitfall 12).
- Zero `@azure/msal-browser` imports across the four new routes (grep verified). Only `login/route.ts` directly imports from `@azure/msal-node` (for `ResponseMode`); `callback/route.ts` goes through `getCca()` from msalClient.

## Decisions Made

- **`ResponseMode` export shape confirmed:** `@azure/msal-node`'s `ResponseMode` is a plain frozen object literal `{ readonly QUERY: "query"; readonly FRAGMENT: "fragment"; readonly FORM_POST: "form_post"; ... }` (verified via `node_modules/@azure/msal-node/dist/index.d.ts`). The plan's mock shape `{ ResponseMode: { QUERY: 'query' } }` matches the runtime export exactly; no surprise. Using `ResponseMode.QUERY` (not FRAGMENT) is load-bearing — fragment is not sent to the server, so the callback handler would have no `?code=` to read.
- **No AADSTS-style error codes surfaced during mock-driven tests:** all Entra-side failure modes were simulated by throwing plain `Error('invalid_grant: state mismatch')` through `acquireTokenByCodeSpy.mockRejectedValue()`. The callback's `catch {}` block treats ANY throw from acquireTokenByCode as "stale code / restart flow", so we do not need to discriminate AADSTS codes here — a future phase can enrich telemetry by reading `err.errorCode` / `err.errorMessage` on msal-node's `AuthError` subclass, but the pilot's response (redirect /api/login) is correct for all of them. No tests needed AADSTS-specific assertions.
- **Vitest top-level-const hoisting bug fixed (Deviations §1):** wrapping spies in forwarding arrow fns `(...args) => spy(...args)` inside vi.mock factories is the fix. The login test file uses the `() => ({ getAuthCodeUrl: getAuthCodeUrlSpy })` pattern which already defers the read via the outer arrow; the callback test file's `{ saveSession: saveSessionSpy }` direct assignment was read at factory-init time, triggering `ReferenceError: Cannot access 'saveSessionSpy' before initialization`. Applied the forwarding-arrow fix to both callback and logout test files.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Fixed Vitest `ReferenceError: Cannot access 'saveSessionSpy' before initialization` in callback test**

- **Found during:** Task 1, first `pnpm test` run after writing the callback test file as-specified in the plan
- **Issue:** The plan's `vi.mock('@/auth/session', () => ({ saveSession: saveSessionSpy }))` factory directly referenced the top-level `const saveSessionSpy = vi.fn()`. Vitest hoists `vi.mock` above all top-level `const` declarations, so the factory ran with the identifier still in the TDZ, throwing `ReferenceError`. The login test file coincidentally escaped this because its factory was `() => ({ getAuthCodeUrl: getAuthCodeUrlSpy })` — the outer arrow deferred the spy read until call time.
- **Fix:** Wrap the spy in a forwarding arrow: `vi.mock('@/auth/session', () => ({ saveSession: (...args: unknown[]) => saveSessionSpy(...args) }))`. Same pattern applied to `/api/logout` test file for `clearSessionSpy`. Added an inline code-comment explaining why.
- **Files modified:** src/app/api/auth/callback/__tests__/route.test.ts, src/app/api/logout/__tests__/route.test.ts
- **Verification:** 9 callback+login tests + 6 logout+me tests all pass.
- **Committed in:** de258b8 (Task 1) and c39091d (Task 2) — the fix was applied BEFORE first commit in both files.
